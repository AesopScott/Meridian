# Relay-Bifrost Proof Payload Contract

**Status:** V0 contract for FileMap registration  
**Date:** 2026-06-01  
**Owner:** Relay (dispatch), Bifrost (display), Build 3 (FileMap registration)  
**Scope:** Stable proof payload keys, immutability, downstream display intent  
**Audience:** Bifrost UI engineers, Build 3, Prime routing

---

## Purpose

Define the stable Relay proof payload structure that Bifrost surfaces can consume without calling Relay internals. Enables Bifrost to display Aegis gate evidence (decision, severity, evidence references, waiver status, explanation) from routing decisions without runtime Relay calls.

---

## Stable Proof Payload Keys

The Relay `AegisGateEvidenceSummary.to_dict()` method serializes to a stable dictionary with these keys:

| Key | Type | Purpose | Immutable? |
|---|---|---|---|
| `gate_decision` | `str \| None` | Gate outcome: `allow`, `demote`, `block`, `human_gate`. | ✓ Yes |
| `severity` | `str \| None` | Evidence severity level: `INFO`, `WARNING`, `ERROR`. | ✓ Yes |
| `evidence_ids` | `tuple[str, ...]` | Ordered tuple of proof evidence IDs from gate validation. | ✓ Yes |
| `waiver_present` | `bool` | Whether a waiver was applied to this gate decision. | ✓ Yes |
| `explanation` | `str` | Explanation text from gate evaluation (empty if none). | ✓ Yes |
| `fallback_blockers_from_aegis` | `tuple[str, ...]` | Ordered tuple of Relay fallback blockers generated by Aegis evidence. | ✓ Yes |

---

## Immutability Contract

All fields in the serialized payload are **immutable types** (or immutable containers):
- `None`, `str`, `bool` are primitive immutable types.
- `tuple` containers are immutable; all elements within tuples are strings (immutable).
- No `list`, `dict`, or custom objects are present.

**Guarantee:** The dictionary structure itself is a plain Python `dict` (mutable container), but all values are immutable. Bifrost may safely read and display values without worrying about in-place mutation.

---

## Determinism and Stability

The serialization is fully deterministic:
- Keys are hardcoded string literals — stable across Python versions and runtimes.
- Key order is preserved (Python 3.7+ guaranteed dict insertion order).
- Values are derived only from frozen `AegisGateEvidenceSummary` instance state; no computed or derived fields.
- Multiple calls to `to_dict()` on the same instance return identical output.

**Guarantee:** Bifrost can safely cache and compare serialized payloads.

## Downstream Display Intent

Bifrost/Prime should display proof payload fields as follows:

### Gate Decision Display
- **Display:** Gate decision as user-readable status label (e.g., "Policy Allowed", "Escalation Required", "Blocked").
- **Mapping:** `allow` → "Allowed", `demote` → "Demoted", `block` → "Blocked", `human_gate` → "Escalation Required".
- **Tone:** Neutral/informational; not a success/failure indicator unless severity warrants it.

### Severity Display
- **Display:** Severity level as a visual indicator (icon/color) or text label.
- **Mapping:** `INFO` → informational (gray), `WARNING` → caution (yellow), `ERROR` → error (red).
- **Tone:** Matches downstream severity; `ERROR` indicates a blocking or critical condition.

### Evidence References
- **Display:** Ordered list of evidence ID references, with optional linking to Aegis proof review surface.
- **Handling:** Empty tuple displays as "no evidence" or omitted.
- **Linking:** If Bifrost has an Aegis surface integration, these IDs should be clickable links.

### Waiver Status
- **Display:** Boolean flag; show "Waiver applied" only if `waiver_present == True`.
- **Tone:** Neutral; acknowledges that a waiver was recorded, not a judgment call.

### Explanation Text
- **Display:** Raw explanation text from gate evaluation.
- **Handling:** Empty string displays as "(no explanation)" or omitted.
- **Formatting:** Plain text; no markup; no interpretation by Bifrost.

### Fallback Blockers from Aegis
- **Display:** Ordered list of Relay fallback blocker strings generated by Aegis evidence (e.g., `aegis_gate_blocked`, `aegis_human_gate_required`).
- **Handling:** Empty tuple displays as "no blockers" or omitted.
- **Tone:** Technical reference; use for Relay operator debugging, not end-user display.

## Out-of-Scope

The following are explicitly out-of-scope and must NOT be done in Bifrost:

- **No Relay runtime calls:** Bifrost does not call `RelayExecutionSummary.aegis_gate_evidence_summary()` or any Relay executor methods. The summary is pre-serialized by Relay and passed to Bifrost.
- **No Aegis calls:** Bifrost does not call Aegis validators, proof checkers, or gate evaluators. Decisions are already made by Relay and Aegis before display.
- **No mutations:** Bifrost does not modify, cache-store, or locally mutate the proof payload. Display only.
- **No UI controls:** Bifrost does not expose gate decision override, waiver creation, or evidence re-evaluation buttons. These belong to the Review Console, not the cockpit.
- **No Bifrost FileMap edits:** Build 1 provides the contract; Build 3 registers it in FileMap. Bifrost does not register new entries.

## JSON Serialization Round-Trip Note

When the proof payload is serialized to JSON (e.g., for API transport or storage):
- `tuple` fields (`evidence_ids`, `fallback_blockers_from_aegis`) serialize as JSON arrays.
- Upon deserialization back to Python, these become `list` (not `tuple`).

**Recommendation for downstream consumers:** If tuple-on-read is required, document this explicitly in the deserialization contract. Treat lists from JSON as equivalent to tuples for display/comparison purposes.

## Testing and Validation

The proof payload serialization is tested with:
- **Empty payload:** Verify all fields have correct empty/None values.
- **Full payload:** Verify all fields preserve data without corruption.
- **Stable keys:** Verify key set consistency across different data values.
- **Immutability:** Verify all value types are immutable.
- **Determinism:** Verify multiple calls return identical output.

See `tests/test_relay_executor.py::TestAegisGateEvidenceSummary` for proof.
